Yes — that’s a very good way to think about it, and it sharpens the picture nicely.

For binary32 (float):
	•	You have 23 explicit mantissa bits
	•	Plus the implicit leading 1 for normal numbers
	•	And the exponent determines the scale, i.e. which power-of-two bin you land in

For numbers in [0,1) specifically, that means:
	•	Values in [0.5,1) have exponent −1
	•	Values in [0.25,0.5) have exponent −2
	•	etc.

So the exponent is effectively acting like an extra “bit” that tells you which half you’re in:
	•	< 0.5 vs ≥ 0.5
	•	then < 0.25 vs ≥ 0.25, and so on

When you generate a float U01 by
u = k * 2^-24,   k ∈ {0,…,2^24−1}
you are really using:
	•	23 bits for the mantissa
	•	1 bit deciding which exponent bucket you’re in (0.5 vs 1.0 scale)

That’s why 24 bits is the correct “native resolution” of a float U01, even though the mantissa is only 23 bits wide.

Why this matters for your exponential case

In the ziggurat expf path:
	•	You explicitly draw only 23 random bits (ri)
	•	The exponent of the float x is then determined deterministically by table geometry (we_float[idx])
	•	So the “extra bit” that normally comes from the exponent bucket is not random

When you later map back via the PIT, that missing randomness shows up as:
	•	collapsed lattice points
	•	doubling of mass at 0
	•	exactly what your counts show

Clean mental model
	•	U01f directly from bits → full 24-bit lattice, uniform
	•	expf ziggurat → PIT → only 23 random bits survive
	•	Endpoint inflation is therefore inevitable, not a bug

Bottom line

Your statement is exactly right:

one can think of one of the exponent bits in the float coding <0.5 vs ≥0.5 so really we have 24 bits there

And the moment you route generation through a path that fixes the exponent structure (like ziggurat tables), you’ve spent that bit.

That’s the correct conceptual resolution of the whole issue.
